> This document is the governance and model card for the Agro-AI credit

## 1. Model Overview

| Field | Value |
|---|---|
| **Model type** | Random Forest Classifier (`sklearn.ensemble.RandomForestClassifier`) |
| **Model version** | `agro-ai-rf-v1` |
| **Task** | Binary classification — predict farmer credit eligibility |
| **Output** | Score (0–100), risk band, eligibility flag, credit limit, top 3 reasons |
| **Estimators** | 160 trees |
| **Max depth** | 7 |
| **Min samples per leaf** | 4 |
| **Class weighting** | Balanced (compensates for class imbalance in training data) |
| **Random seed** | 42 (fully reproducible) |

## 2. Training Data

### 2.1 Current state — synthetic
The model is currently trained on **`DEMO_FARMERS`** — a deterministic
synthetic dataset generated by `backend/app/agro_ai/synthetic_data.py`.

This dataset is designed to demonstrate the pipeline and produce realistic
score distributions. It is **not** derived from real farmer records.

**What this means:**
- Evaluation metrics (ROC AUC, F1, etc.) measured on this dataset prove the
  model can approximate the synthetic label rule — not that it generalises to
  real cooperative data
- The model must not be presented as a production-grade credit risk tool until
  retrained on real outcomes

### 2.2 Features used

The model takes six numerical features per farmer:

| Feature | Description |
|---|---|
| `dues_payment_rate` | Proportion of cooperative dues paid on time |
| `on_time_payment_rate` | General on-time payment rate across obligations |
| `yield_performance` | Farm harvest output relative to cooperative benchmark |
| `attendance_rate` | Cooperative meeting attendance rate |
| `outstanding_balance_ratio` | Current outstanding balance as a proportion of credit limit |
| `savings_rate` | Savings consistency over the assessment period |

### 2.3 Label definition
Training labels are the `eligible` boolean — derived from the synthetic
generator's rules. In production this must be replaced with real repayment
outcomes from Supabase and Moolre transaction history.

## 3. Score Thresholds and What They Mean

The model outputs a score from 0–100 (probability × 100). Three thresholds
govern lending decisions:

| Score range | Risk band | Recommendation |
|---|---|---|
| **82 – 100** | Low risk | Approve full input credit |
| **68 – 81** | Moderate risk | Approve with standard monitoring |
| **55 – 67** | Watchlist | Review manually before approval |
| **0 – 54** | High risk | Defer credit and require dues recovery |

**Eligibility threshold:** score ≥ 68. Farmers below this threshold are not
automatically approved; their credit limit is capped at 55% of the requested
amount.

**Credit limit formula:**
```
credit_limit_factor = 0.45 + (score / 100)
approved_credit_limit = round(requested_amount × credit_limit_factor / 50) × 50
approved_credit_limit = min(approved_credit_limit, requested_amount)

# If not eligible:
approved_credit_limit = min(approved_credit_limit, requested_amount × 0.55)
```

## 4. Explainability — Top Reasons

Each prediction includes three human-readable reasons drawn from the six
features. The reasons are ranked by how far each feature deviates from a
neutral threshold of `0.62` — features furthest from neutral have the
greatest influence on the score.

Example reasons:
- *"Strong dues consistency improves repayment confidence"*
- *"Yield performance is below the cooperative benchmark"*
- *"Low outstanding balance leaves room for new credit"*

> ⚠️ These reasons are heuristic approximations, not SHAP or LIME explanations.
> Production lending decisions should use a reviewed explainability approach
> and a formal model card. See `docs/agro-ai-evaluation.md`.

## 5. Audit Log

Every prediction made through `POST /api/agro-ai/predict` is written to:

```
backend/logs/agro_ai_predictions.jsonl
```

Each log entry captures:
- `farmer_id`, `cooperative_id`, `actor_id` (when provided by the caller)
- Input features
- Score, risk band, eligibility, approved credit limit
- Model version and feature schema version
- Timestamp

In the current hackathon build these fields are optional — the endpoint
accepts them but does not require authentication. In production, auth will
populate `actor_id` and `cooperative_id` from the session without changing
the model contract.

## 6. Not a Substitute for Cooperative Judgment

> ⚠️ **Important disclaimer for demo and hackathon use.**

The Agro-AI score is a **decision-support tool**, not an autonomous lending
system. It is designed to surface relevant signals to cooperative
administrators — not to replace their judgment.

**During the hackathon demo:**
- All scores are computed from synthetic data
- No real farmer's creditworthiness is being assessed
- Scores should not be used to make any real financial decisions

**In production:**
- A cooperative leader or admin must review and approve any credit decision
- A farmer may not be denied credit solely on the basis of an Agro-AI score
- The manual review pathway (score 55–67, Watchlist) must be staffed and functional
- Farmers must be informed if a score is used in a decision affecting them
  (see `docs/data-privacy.md`)

## 7. Trust Score vs Agro-AI — When to Use Which

Two scoring systems run in parallel. This is a summary; see
[`docs/scoring-systems.md`](scoring-systems.md) for the full breakdown.

| Situation | Use |
|---|---|
| Farmer pays dues via USSD and checks their score | **Trust Score** (rules-based, DB-backed) |
| Cooperative admin reviews dashboard credit profile | **Agro-AI** (ML, synthetic data in demo) |
| Webhook fires after Moolre payment | **Trust Score** recalculates |
| Demo narration of ML capability | **Agro-AI** |
| Production credit decision (future) | **Unified score** (see Roadmap) |

## 8. Promotion Path — Synthetic to Production

The model is ready for promotion when the following conditions are met:

### Phase 1 — Real labels (post-hackathon)
- [ ] Replace `DEMO_FARMERS` with Supabase-backed farmer records
- [ ] Define repayment outcome labels from Moolre transaction history
      (dues repayment, loan repayment, late repayment, default)
- [ ] Retrain and re-evaluate against real outcomes
- [ ] Document updated ROC AUC, PR AUC, and calibration in
      `docs/agro-ai-evaluation.md`

### Phase 2 — Governance (pre-production)
- [ ] Replace heuristic explanations with SHAP or LIME
- [ ] Complete a bias audit across cooperative regions and crop types
- [ ] Establish a model registry and promotion review process
- [ ] Require two-person review for threshold changes
- [ ] Publish a revised model card approved by the team

### Phase 3 — Monitoring (production)
- [ ] Persist audit logs to Supabase with tenant and actor IDs
- [ ] Set up score distribution drift monitoring
- [ ] Alert on latency or error rate anomalies
- [ ] Schedule periodic model retraining from accumulated DB history

## 9. Known Limitations

| Limitation | Impact | Mitigation |
|---|---|---|
| Trained on synthetic data only | Scores do not reflect real farmer risk | Do not use for real decisions; label clearly in demo |
| Heuristic explainability | Reasons may not reflect true feature importance | Flag for SHAP/LIME in Phase 2 |
| No bias audit | Unknown performance across demographic subgroups | Required before production |
| No model registry | Artifact promotion is manual | CI-based promotion in Phase 2 |
| Threshold values are fixed | Cooperative risk appetite may vary | Admin-tunable thresholds in roadmap |
| No auth on predict endpoint | Any caller can log any actor_id | Auth integration tracked separately |
