# Security Policy

## Reporting a Vulnerability

If you discover a security vulnerability in AgroOS, please do not open a
public GitHub issue. Contact the maintainers directly via the Moolre
Startup Cup hackathon communication channels (team email — TBD; update
this section once a dedicated contact is available).

We will acknowledge receipt within 48 hours and aim to resolve confirmed
issues before any public demo or production deployment.

## Scope

During the hackathon phase (Moolre Startup Cup, July 2026), the following
areas are in scope for security review:

- FastAPI backend endpoints (authentication, input validation)
- Supabase row-level security configuration
- Moolre webhook signature verification
- Environment variable and secret handling (`.env.example` hygiene)
- Third-party dependency management

## Secrets Management

- All secrets live in `.env` files, which are gitignored at the repo root
  and in `backend/`. `.env` is never committed.
- `backend/.env.example` and the root `.env.example` are templates only —
  they contain empty or placeholder values and must be copied to `.env`
  locally (`cp backend/.env.example backend/.env`) rather than edited in
  place.
- Do not commit real Supabase keys, Moolre API credentials, or the
  `MOOLRE_WEBHOOK_SECRET` to any file, commit message, issue, or log.

## Moolre Webhooks

AgroOS exposes two Moolre webhook routes (`backend/app/routes/webhooks.py`):

- **`POST /webhooks/moolre/payment`** — verified. Every request's HMAC-SHA256
  signature (`X-Moolre-Signature` header) is checked against a digest
  computed with `MOOLRE_WEBHOOK_SECRET` before the payload is processed.
  Unsigned or invalid requests are rejected with `401 Unauthorized`.
- **`POST /webhooks/moolre/ussd`** — **not signed today.** The USSD session
  handler has no signature or origin check, so any client that can reach
  the endpoint can drive USSD menu logic. This is acceptable for the
  hackathon demo but is a known gap that must be closed (e.g. shared
  secret, IP allowlist, or Moolre-provided signing once available) before
  production use.
- `MOOLRE_WEBHOOK_SECRET` is **required in production**. If it is unset,
  `_verify_signature()` logs a warning and skips verification entirely —
  this dev-only fallback exists purely so the payment webhook can be
  exercised locally without live Moolre credentials, and must never be
  relied on outside local development.

## Public API Surface

Authentication has not been implemented yet. Every route under
`backend/app/routes/` (cooperatives, farmers, transactions, loans,
production, communications, and the AgroCredit AI endpoints) is currently
reachable without credentials. Treat any deployment of this codebase as
publicly readable/writable until auth and RBAC land, and do not point it
at real farmer data or real Moolre production credentials in the interim.

## Dependencies

- Backend dependencies are pinned via `pip-compile` from
  `backend/requirements.in` into `backend/requirements.txt`. Hash pinning
  (`pip-compile --generate-hashes`) is not yet enabled — this is a
  recommended follow-up so `pip install` can verify package integrity,
  not just version.
- JWT handling uses `PyJWT[crypto]`, not `python-jose`, specifically to
  avoid the algorithm-confusion and CVE history associated with
  `python-jose`. Do not reintroduce `python-jose` as a dependency.
- Dependency updates should go through `requirements.in` followed by
  `pip-compile requirements.in`, not by hand-editing `requirements.txt`.

## Data Privacy (PII)

AgroOS collects and processes farmer PII including names, phone numbers,
financial transactions, credit scores, and SMS content. Farmer phone
numbers are stored both in the database (`Farmer.phone`) and in SMS
delivery logs generated by the communications service. For the full data
handling policy, PII categories, access scope, SMS consent requirements,
and retention schedule, see [docs/data-privacy.md](docs/data-privacy.md).

## Known Limitations (Hackathon Phase)

- No authentication or RBAC is enforced in the current demo build.
- The USSD webhook (`/webhooks/moolre/ussd`) is unsigned; only the payment
  webhook verifies signatures.
- The system must only be run with synthetic demo data until auth is
  implemented.
- No rate limiting is currently applied to API endpoints.
- Dependency hashes are not yet pinned.
